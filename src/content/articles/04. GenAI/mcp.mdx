# Model Context Protocol (MCP)

Das **Model Context Protocol (MCP)** ist ein offener Standard, der die Art und Weise vereinfacht, wie KI-Modelle auf externe Systeme zugreifen. Entwickelt von Anthropic, zielt MCP darauf ab, die Integration von Tools, Datenquellen und Software-Schnittstellen (APIs) zu standardisieren.

---

## Das Problem vor MCP

:::beginner
Bevor es MCP gab, musste für jedes KI-Tool und für jede Datenquelle (wie lokale Dateien, GitHub, Datenbanken oder Notion) eine eigene Brücke gebaut werden. Das bedeutet: Wenn ein Entwickler wollte, dass ChatGPT seine Dateien lesen kann, musste er einen speziellen Adapter schreiben. Wollte er dasselbe für Claude tun, musste er wieder von vorn anfangen.
:::

:::medium
Ohne einen einheitlichen Standard entwickelte das Ökosystem sich als "Plugin-Wüsten". Jedes LLM (OpenAI, Anthropic, Google) brachte sein eigenes proprietäres Tool-Calling-Format mit. Die Konsequenz: Tool-Entwickler mussten den Code N-mal neu schreiben, pflegen und anpassen, um verschiedene Modelle zu unterstützen. Dies hemmte die Verbreitung von leistungsstarken, modellübergreifenden Werkzeugen.
:::

:::advanced
Vor MCP bestanden Tool-Calling-Architekturen typischerweise aus proprietären JSON-RPC Wrappern. Bei der Integration von externen Systemen mussten Model-Provider oft als Middleman agieren oder Entwickler mussten Middleware-APIs betreiben, um das Custom-Routing zu verwalten. Dies erschwerte auch lokales Development erheblich, da eine saubere Trennung zwischen LLM-Client (der die Instruktionen und Prompt-Chains steuert) und dem Tool-Executor (der die tatsächliche Sandbox/Datenbank kontrolliert) oft fehlte.
:::

---

## Wie funktioniert MCP?

:::beginner
**MCP** fungiert wie ein universeller Stecker.  
Stell dir vor: Anstatt für jedes Land einen anderen Steckdosen-Adapter zu brauchen, gäbe es weltweit nur noch einen Stecker. Das LLM spricht eine universelle Sprache, um mitzuteilen: *"Ich brauche diese Daten"* oder *"Führe diese Aktion aus"*. Der MCP-Server hört zu, erledigt die Aufgabe vor Ort auf deinem Computer und gibt dem Modell die Antwort zurück.
:::

:::medium
Die MCP-Architektur basiert auf einer strikten Client-Server-Trennung:
1. **MCP Host (Client):** Eine App wie Claude Desktop, dein Code-Editor oder Cursor, welche die LLM-Verbindung managt.
2. **MCP Client:** Der Code im Host, der mit dem Server via JSON-RPC kommuniziert.
3. **MCP Server:** Das Programm (lokal oder remote), das den tatsächlichen Zugang zu Ressourcen, Prompts und Tools anbietet.

Diese Struktur wird über sicheres **stdio** (Standard Input/Output) im lokalen System oder über **SSE** (Server-Sent Events) bei Remote-Setups abgewickelt.
:::

:::advanced
MCP ist technologisch nichts anderes als ein standardisiertes JSON-RPC 2.0-Protokoll mit einem definierten Set an Features (Capabilities). Die wichtigsten sind:
- **Resources:** Lese-Zugriff auf Kontext wie Dateien, Logs oder Datenbank-Ergebnisse (Ressourcen sind typischerweise stateful URIs).
- **Prompts:** Vordefinierte System-Prompts, die eine UI dem Nutzer zur Verfügung stellen kann.
- **Tools:** Funktionen, die das LLM ausführen kann. Der Server beschreibt die Parameter per JSON-Schema.

Eine Transaktion läuft asynchron: Der Client ruft `tools/call` auf, der Server validiert den Request, führt ihn in seiner isolierten Sandbox aus und sendet `ToolResult` textuell oder binär zurück.
:::

## Sicherheit und reale Anwendungen

Da MCP lokalen Systemzugriff ermöglicht, ist das Thema **Sicherheit** zentral im Architektur-Design verankert:
- **Lokale Ausführung:** Tools laufen standardmäßig als lokale Subprozesse, weshalb sensible Firmendaten nie über fremde APIs geroutet werden müssen, wenn das LLM ebenfalls lokal läuft.
- **Opt-in-Prinzip:** Moderne MCP Hosts erzwingen oft eine explizite Bestätigung durch den Nutzer, bevor ein Tool ausgeführt wird. 
- **Zwei-Wege-Sicherheit:** Weder der Client noch der Server kennen die internen Speicher des jeweils anderen. Sie kommunizieren rein über JSON-RPC Grenzen hinweg.

<InfoBox type="note" title="Wer nutzt MCP bereits?">
  Zahlreiche Tools unterstützen MCP bereits nativ. **Claude Desktop** und fortschrittliche IDEs wie **Cursor** nutzen MCP, um kontext-aware auf das Dateisystem und Terminal zuzugreifen. Firmen können so eigene firmeninterne APIs extrem schnell an diese Tools andocken.
</InfoBox>

## Implementierung mit FastMCP

Da das manuelle Parsen von JSON-RPC-Nachrichten komplex ist, stellt Anthropic das offizielle **MCP Python SDK** bereit. Das Herzstück für Entwickler ist dabei **FastMCP** — ein High-Level Framework, das sich ähnlich wie FastAPI oder Flask anfühlt.

### Beispiel in Python (FastMCP)

Mit FastMCP lassen sich in Python reguläre Funktionen mit simplen Decorators in vollständige MCP-Tools verwandeln.

```python
from mcp.server.fastmcp import FastMCP
import sqlite3

# 1. Initialisiere den MCP Server (läuft standardmäßig auf stdio)
mcp = FastMCP("Employee DB")

# Verbinde zu einer internen Datenbank
conn = sqlite3.connect("company.db")

# 2. Registriere ein Tool über den Decorator
# Der Docstring und die Type-Hints werden automatisch vom LLM gelesen!
@mcp.tool()
def search_employee(name: str) -> str:
    """
    Sucht nach Mitarbeitern in der Unternehmensdatenbank.
    Diese Beschreibung sieht das LLM, um zu verstehen, was das Tool macht.
    """
    cursor = conn.cursor()
    cursor.execute("SELECT name, department, email FROM employees WHERE name LIKE ?", ("%" + name + "%",))
    results = cursor.fetchall()
    
    if not results:
        return "Keine Mitarbeiter mit dem Namen '" + name + "' gefunden."
        
    # Formatiere das Ergebnis als Text, den das LLM gut verarbeiten kann
    return "\n".join(["- " + r[0] + " (Abteilung: " + r[1] + ") | Email: " + r[2] for r in results])

# 3. Starte den Server
if __name__ == "__main__":
    mcp.run()
```

Wenn ein MCP Host (wie z. B. Claude Desktop) diesen Server per Command-Line startet (`python server.py`), fragt er zunächst via `tools/list` die Fähigkeiten ab. Das Script liefert automatisch die Parameter (`name: str`) und den Docstring als Tool-Beschreibung an das LLM. Das LLM kann anschließend selbstständig die Datenbank abfragen, ohne dass du Prompt-Engineering im Code betreiben musst.
